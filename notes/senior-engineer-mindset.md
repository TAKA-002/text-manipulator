# 上級エンジニアのタスクへの向き合い方

## 前提

上級者が速い理由は「才能」ではなく、**経験の量と質**。
同じところで詰まった経験の積み重ねがあるから速い。

---

## 思考の「型」5つ

### 1. 作る前に「分解」する

手を動かす前に、問題を小さく切り分けることに時間を使う。
手を動かし始める時点で、頭の中に完成形がほぼ見えている。

```
「大文字変換を作る」ではなく
  → ロジックは何か？
  → オプションは必要か？
  → state は必要か？
  → 既存のパイプラインのどこに入れるか？
```

### 2. 「既存のパターンに乗れるか」を先に探す

ゼロから考えない。まずコードベースを見て、似たものを探す。

> 「あのコンポーネントと同じ構造で作れそう」→ それをベースに書く

新しいコードを書くより、既存パターンを流用する方が速くバグも少ない。

### 3. 「今解かなくていい問題」を意図的に後回しにする

気になったことを全部その場で解決しようとしない。
スコープを意識的に絞る。

> 「エラーハンドリングは後でいい、まず動かす」
> 「UIの細かいデザインは機能が確定してから」

### 4. 動くものを最速で作って、そこから育てる

完璧なものをいきなり作ろうとしない。

```
汚くてもいいから動くものを作る
  → 動作確認する
  → 整理する
```

最初から綺麗に書こうとすると詰まる。この順番を守る。

### 5. 詰まったら「手を動かすのをやめる」

詰まった瞬間に一度止まって、**「何がわかっていないか」を言語化する。**

> 「useEffect が2回走るのはなぜか」がわかっていない
> → 調べるべき問いが明確になる

闇雲に試行錯誤する時間が少ない。

---

## まとめ

上級者がやっていることは特別なことではない。
この5つを**無意識にできるようになった状態**にすぎない。

最初は意識的にやるしかないので、新機能を実装するときに
手を動かす前に上の5ステップで考えてみることを繰り返す。
